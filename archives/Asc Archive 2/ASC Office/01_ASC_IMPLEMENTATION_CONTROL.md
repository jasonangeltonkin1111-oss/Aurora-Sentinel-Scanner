## 4. Archive and lineage usage law

### 4.1 Archive role
The archive is reference and lineage evidence.
It is not active runtime or office canon.

### 4.2 Translation states
Every archive use must be classified as:
- **PRESERVE** — keep the core truth almost directly, translated into ASC form
- **ADAPT** — preserve the principle but change structure or wording materially
- **REJECT** — keep only as a warning or provenance artifact

### 4.3 Current lineage decisions
#### AFS
- preserve: bounded rotation, bucket honesty, HUD truthfulness, pending/stale distinctions, summary-last discipline
- adapt: dossiers into canonical symbol continuity files; stage language into runtime-domain language
- reject: HUD/dev-mode sprawl and broad downstream theater

#### EA1

#### Archived ASC blueprint/office
- preserve: layer separation, activation boundary, writer purity, UI boundary, `PrimaryBucket` authority, archive discipline
- adapt: old milestone language into refounded implementation sequencing

## 5. Current contradiction register

This register is the planning bridge between blueprint truth and implementation reality.

### C1. Init is too feature-heavy
Observed reality:
- the archived MT5 `OnInit()` enables a large feature/config surface immediately
- init effectively authorizes surface work, publication options, mirror output, HUD/menu, and many toggles before a real kernel shell exists

Required resolution:
- implementation must create a shell-first init
- later features activate only when earlier runtime stages exist

Status: OPEN UNTIL STAGE 1 IMPLEMENTATION.

### C2. Runtime is still single-pass collapsed
Observed reality:
- archived engine pass restores, discovers, computes, saves snapshot, and publishes inside one control path

Required resolution:
- split into due services under kernel control

Status: OPEN UNTIL STAGES 1 TO 6.

### C3. Publication is too close to truth mutation
Observed reality:
- save/publish occurs directly after processing within the same pass

Required resolution:
- publication becomes a separate commit service with gating and reserved budget

Status: OPEN UNTIL STAGE 5/6 FOUNDATION.

### C4. HUD still leaks implementation/control language shape
Observed reality:
- runtime phase text and broad config menus act like build/control rails rather than a bounded operator/trader surface

Required resolution:
- final HUD uses prepared runtime read models and domain language only

Status: OPEN UNTIL FINAL UI STAGE.

### C5. Restore-first exists, but restore constitution is incomplete in implementation
Observed reality:
- archived runtime restores snapshot continuity, but lacks the full runtime mode / journal / compatibility / recovery-hold constitution now required

Required resolution:
- implement restore as a first-class runtime mode with honest outcomes

Status: OPEN UNTIL STAGE 1 + 5.

### C6. Whole-universe, promoted-set, and summary boundaries are still too easy to collapse
Observed reality:
- archived systems and runtime patterns still risk blurring publication scope, active scope, and broad universe truth

Required resolution:
- implementation must keep separate in-memory and published representations for universe, promotion, and summary

Status: OPEN UNTIL STAGES 4 TO 8.

### C7. Office truth was historically too scattered
Observed reality:
- the archived office layer contains many useful but overlapping control and handoff documents

Required resolution:
- active office canon is now this file plus the stage execution plan only

Status: CLOSED IN DOCUMENTATION; ENFORCE GOING FORWARD.

## 6. Review and handoff law

### 6.1 What every implementation slice must prove
Every slice must prove:
- which blueprint section it fulfills
- which contradictions it retires
- what new runtime truth became explicit
- what remains intentionally unimplemented
- what tests/checks were run
- what failure cases remain open

### 6.2 Handoff format
Any future handoff should be short and structured:
1. slice objective
2. blueprint sections touched
3. contradictions retired
4. runtime functions added or clarified
5. tests/checks
6. residual open risks

No new sprawling wave reports by default.

### 6.3 Review focus
Review must check for:
- boundary collapse
- hidden later-stage assumptions
- UI taking logic ownership
- publication bypassing validation order
- whole-universe amnesia
- dev-language leaks into product surfaces
- borrowed legacy behavior without translation

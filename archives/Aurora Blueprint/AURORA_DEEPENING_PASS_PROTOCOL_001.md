# AURORA DEEPENING PASS PROTOCOL 001

## PURPOSE

This file defines how future Aurora passes must go materially deeper than earlier extraction or first-consolidation passes.

It exists because the project has now crossed the stage where thin passes, light summaries, and first-order preservation are enough.

From this point onward, every meaningful pass should aim to increase one or more of:
- boundary sharpness
- discriminating power
- operator usability
- wrapper / EA translation readiness
- explicit negative-space mapping

This protocol does not replace source discipline or lineage law.
It strengthens what a serious pass must deliver.

---

# 1. ROOT LAW

A deeper pass is not a longer pass.

A deeper pass is a pass that produces more decision-useful structure.

That means a valid deeper pass should do at least one of:
- distinguish similar things that should not be treated as the same
- separate valid use from invalid use
- make suppression / downgrade / escalation boundaries clearer
- make translation into evidence, ranking, wrapper logic, or EA logic easier
- expose where a concept fails, not only where it works

If a pass only restates what already exists in cleaner language, it is not deep enough.

---

# 2. WHAT “GO DEEPER” MEANS

Future passes should aim to add one or more of these layers:

## 2.1 Boundary layer
Clarify:
- where the concept begins
- where it stops
- what it is often confused with
- what should be explicitly excluded

## 2.2 Failure layer
Clarify:
- how the concept fails
- what naive use looks like
- what breaks when it is misapplied

## 2.3 Translation layer
Clarify:
- how the concept maps into evidence notes
- wrapper prompts
- ranking logic
- suppression logic
- EA-safe inputs later

## 2.4 Competition layer
Clarify:
- what alternative explanations compete with this one
- what evidence would favor one over the other

## 2.5 Escalation layer
Clarify:
- when a mild condition becomes severe
- when caution becomes suppression
- when observation becomes prohibition

## 2.6 Cross-wave interaction layer
Clarify:
- how the concept should interact with Wave 1 structure
- Wave 3 evaluation
- Wave 4 hostility
- Wave 5 review/adaptation

---

# 3. DEEPENING PASS MINIMUM REQUIREMENT

Every serious pass after this protocol should produce at least three of the following:

1. one sharper distinction
2. one explicit anti-confusion rule
3. one negative-space or failure-mode mapping
4. one translation hook into wrapper / ranking / EA-safe logic
5. one escalation boundary
6. one cross-wave interaction rule

If fewer than three are added, the pass is probably too thin.

---

# 4. WHAT DEEPER PASSES MUST NOT BECOME

Deeper passes must not become:
- bloated prose
- vague “more nuance” without boundaries
- speculative expansion unsupported by repo truth
- complexity for its own sake
- hidden redefinition of existing doctrine

Depth without sharper utility is false depth.

---

# 5. DEEPENING ORDER

The preferred order for deeper work after first consolidation is:

1. refresh control truth so new consolidated surfaces are canonical
2. deepen Waves 3–5 using sharper boundary / translation / escalation passes
3. open Wave 2 consolidation only once the deeper-pass rule is active
4. later connect the consolidated waves into wrapper / ranking / EA-safe translation layers

---

# 6. PASS OUTPUT EXPECTATION

A deeper pass should usually create or strengthen one of these:
- consolidated doctrine addendum
- deeper doctrine strengthening file
- translation map
- anti-confusion map
- escalation map
- wrapper-alignment note

---

# 7. CURRENT JUDGMENT

Aurora is now at the stage where every pass should do more than preserve truth.
It should sharpen utility.

This protocol exists to force that change.

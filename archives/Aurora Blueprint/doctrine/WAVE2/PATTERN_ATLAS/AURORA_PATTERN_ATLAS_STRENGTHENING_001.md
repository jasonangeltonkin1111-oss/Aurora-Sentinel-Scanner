# 2. PATTERN-ATLAS STRENGTHENING ADDITIONS

## 2.1 Taxonomy-first rule

### Strengthening statement
Aurora’s Pattern Atlas should be taxonomy-first and context-first before it becomes statistics-first or tactics-first.

### Why this is justified
Bulkowski organizes patterns comprehensively, while Grimes and Murphy embed patterns inside structure and context. Nison adds naming nuance but not a complete market-state doctrine.

### Canonical implication
Each pattern entry should first preserve:
- canonical name
- aliases
- family relationship
- state habitat
- location/context requirement
- common failure mode

---

## 2.2 Continuation-pattern bucket

### Strengthening statement
Continuation patterns should be a major atlas bucket, but they must inherit from trend or directional habitats rather than be treated as context-free shapes.

### Source support
- Murphy’s classic continuation-pattern framing
- Bulkowski’s catalog support
- Grimes’ trend and pullback structures

### Atlas implication
Continuation entries should record:
- trend habitat required
- typical pullback/base structure
- common failure form

---

## 2.3 Reversal-pattern bucket

### Strengthening statement
Reversal patterns should be grouped as transition structures, not just “opposite direction” drawings.

### Source support
- Murphy’s classical reversal vocabulary
- Bulkowski’s reversal catalog
- Nison’s candlestick reversal naming
- Grimes’ trend-to-range and trend-to-opposite-trend interfaces

### Atlas implication
Reversal entries should record:
- prior state required
- transition quality
- common false-positive environment

---

## 2.4 Breakout-pattern bucket

### Strengthening statement
Breakout patterns belong as their own bucket, but the atlas must distinguish:
- genuine breakout / release
- weak breakout attempt
- failed breakout

### Source support
- Grimes’ breakout trade and failed breakout logic
- Bulkowski’s broad breakout-style pattern catalog
- Murphy’s classical pattern framing

### Atlas implication
Breakout entries should preserve both success path and failure path.

---

## 2.5 Pullback-structure bucket

### Strengthening statement
Pullback structures deserve a dedicated bucket rather than being hidden under generic continuation.

### Source support
- Grimes makes pullbacks central
- Murphy supports trend continuation context

### Atlas implication
Pullback entries should record:
- directional habitat required
- retracement quality
- what turns a pullback into a failure or transition instead

---

## 2.6 Candlestick-pattern bucket

### Strengthening statement
Candlestick patterns should be included as a dedicated atlas bucket, but treated as pattern language and nuance support, not as standalone complete strategy doctrine.

### Source support
- Nison’s candlestick taxonomy

### Atlas implication
Candlestick entries should record:
- pattern name
- common alias if any
- reversal or continuation tendency
- habitat where the signal has meaning
- common misuse risk

### Main anti-error
Do not let candlestick labels replace market-state or family context.

---

## 2.7 Trading-range pattern bucket

### Strengthening statement
Trading-range structures should be preserved as pattern objects because ranges are functional structures, not just a lack of trend.

### Source support
- Grimes on trading ranges as functional structures
- Murphy support/resistance framing
- Bulkowski rectangle and related patterns

### Atlas implication
Entries should record:
- range habitat
- edge vs center behavior
- breakout / failure possibilities

# 3. FAILURE PRESERVATION RULE

## 3.1 Every major pattern bucket should preserve failure forms

### Strengthening statement
Aurora should preserve failure as part of pattern doctrine rather than treat it as an afterthought.

### Why this is justified
Bulkowski and Grimes both strongly preserve failure structures.

### Atlas implication
Each major pattern bucket should explicitly track:
- what failure looks like
- when the pattern degrades into trap logic
- when the pattern implies transition rather than continuation

---

# 4. NAMING AND ALIAS RULE

## 4.1 Multiple naming traditions must be preserved

### Strengthening statement
The atlas should preserve multiple naming traditions rather than flatten everything prematurely into one title.

### Why this is justified
Murphy, Bulkowski, Grimes, and Nison all contribute naming systems with overlap and differences.

### Atlas implication
Each pattern entry should preserve:
- canonical Aurora name
- source aliases
- category bucket

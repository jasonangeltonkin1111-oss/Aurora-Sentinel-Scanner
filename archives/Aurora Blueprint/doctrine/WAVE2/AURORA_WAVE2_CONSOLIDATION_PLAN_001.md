# AURORA WAVE 2 CONSOLIDATION PLAN 001

## PURPOSE

This file opens the formal consolidation-planning phase for Wave 2.

Wave 2 is now the richest unconsolidated doctrine lane in Aurora.
It contains:
- strategy families
- setup patterns
- family cards
- pattern cards
- evidence layer
- quant / machine lane
- supportive expansions from Source Sets 008 and 010

This also makes Wave 2 the highest-risk consolidation lane.
If consolidated too quickly or too bluntly, it will flatten distinctions that Aurora still needs.

This plan exists to ensure that Wave 2 consolidation follows the deepening-pass protocol rather than becoming a summary dump.

---

# 1. ROOT LAW

Wave 2 consolidation must increase discrimination, not reduce it.

That means a valid Wave 2 consolidation pass must:
- sharpen family boundaries
- sharpen pattern boundaries
- preserve anti-confusion rules
- map suppression and misuse zones
- add translation hooks into evidence / wrapper / EA-safe logic later

If a Wave 2 pass only collects names and examples into one file, it has failed.

---

# 2. WHAT MAKES WAVE 2 HARD

Wave 2 is hard because it contains multiple interacting layers:
- abstract family classes
- pattern vocabularies
- cards and evidence notes
- discretionary structure interpretations
- quant / machine translation possibilities
- supportive reflexivity / stat-arb / ML realism

This means the main risk is not missing content.
The main risk is false flattening.

---

# 3. REQUIRED CONSOLIDATION ORDER INSIDE WAVE 2

## Phase A — Strategy-family system first

Primary target:
- `Aurora Blueprint/doctrine/WAVE2/AURORA_STRATEGY_FAMILY_SYSTEM_WAVE2_CONSOLIDATED.md`

Why first:
- families are the highest-order organizing layer
- patterns, cards, and evidence should hang from family logic, not override it

Minimum deepening requirements:
- sharper family boundaries
- anti-confusion rules between adjacent families
- habitat / anti-habitat mapping
- failure / misuse mapping
- translation hooks into evidence and quant-machine lanes

## Phase B — Pattern doctrine second

Primary target:
- `Aurora Blueprint/doctrine/WAVE2/AURORA_PATTERN_SYSTEM_WAVE2_CONSOLIDATED.md`

Why second:
- pattern language should be subordinated to family logic
- pattern consolidation before family consolidation increases flattening risk

Minimum deepening requirements:
- pattern-group boundaries
- family-to-pattern linkage rules
- anti-pattern / false-positive mapping
- measurable-hypothesis hooks

## Phase C — Quant / machine consolidation third

Primary target:
- `Aurora Blueprint/doctrine/WAVE2/AURORA_QUANT_MACHINE_WAVE2_CONSOLIDATED.md`

Why third:
- the machine lane should formalize and test the family/pattern stack, not define it alone

Minimum deepening requirements:
- structure-vs-ML boundaries
- complexity-admission rules
- evidence-translation hooks
- expected-return and implementation continuity

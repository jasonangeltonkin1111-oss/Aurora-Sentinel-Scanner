AFS HUD MASTER BLUEPRINT
Project Track: HUD / Visibility Architecture
Status: Side Project Blueprint
Scope: Long-horizon staged HUD overhaul
Applies To: Aegis Forge Scanner (AFS) MT5 EA
Priority: Visibility, truthfulness, operator usability, non-disruptive integration

============================================================
1. PURPOSE
============================================================

This blueprint defines the long-term HUD architecture for AFS.

The purpose is to separate:
- operator/developer scanner visibility
from
- trader-facing symbol/bucket visibility

The HUD must become truthful, progress-aware, and dependency-safe.

The HUD must not guess.
The HUD must not compute heavy logic itself.
The HUD must not become a runtime stall point.
The HUD must render already-known scanner truth clearly.

Core intent:
- show what is already known instantly
- show what is pending
- show why it is pending
- show progress at bucket level
- show dossier/file completion state
- keep trader-facing HUD clean
- support future click navigation safely over time

============================================================
2. CORE HUD SPLIT
============================================================

AFS will use two distinct HUD tracks:

A. DEV HUD
Purpose:
- scanner/developer/operator visibility
- file-write visibility
- publication progress
- bucket completion
- pending reasons
- batch rotation status
- dossier completion truth
- runtime progress without opening files

B. TRADER HUD
Purpose:
- trader-facing package view only
- active buckets
- symbols inside buckets
- later drilldown into bucket symbols
- later drilldown into symbol signal views
- no heavy internal scanner diagnostics

Rule:
- Dev HUD and Trader HUD must not be blended again
- Dev HUD is operational
- Trader HUD is trader-facing

============================================================
3. PRIMARY DESIGN PRINCIPLES
============================================================

1. Scanner computes, HUD displays
2. HUD must consume existing runtime state only
3. HUD must never become a new scanner engine
4. HUD must not change selection truth
5. HUD must not change correlation truth
6. HUD must not change publication truth
7. HUD must not re-query heavy broker data every frame
8. HUD must not iterate files expensively every repaint
9. HUD must remain readable for a visually constrained operator
10. HUD must emphasize completeness and pending causes
11. HUD must be stage-aware, bucket-aware, and symbol-aware
12. HUD must remain safe under partial data availability
13. HUD must make “known now vs pending later” obvious
14. HUD must support staged implementation
15. HUD buttons must be simplified and stabilized before expansion

============================================================
4. WHAT THE HUD MUST SOLVE
============================================================

Current pain points:
- files may look incomplete even when much data already exists
- operator cannot easily see what has been written
- operator cannot easily see which buckets are complete
- operator cannot easily see which symbols are pending and why
- trader HUD currently shows too much internal detail
- button behavior is unreliable and should not be overcomplicated
- visibility into rotation / batch progress is insufficient
- operator needs completion percentages and concrete file status
- scanner truth and dossier truth are not surfaced clearly enough

Target outcome:
- immediate visibility into what data is already present
- explicit visibility into what data is still pending
- bucket-level progress summaries
- symbol-level readiness classification
- clean trader-facing bucket/symbol HUD
- future-safe drilldown design

============================================================
5. DATA CATEGORIES
============================================================

The HUD and dossier system must distinguish four classes of information.

A. STATIC / BROKER / SYMBOL SPEC DATA
This should appear instantly if known:
- symbol
- canonical
- display name
- asset
- bucket
- digits
- point
- tick size
- tick value
- contract size
- volume min/max/step
- stops level
- freeze level
- calc mode
- currency fields
- broker spec data
- server/symbol specification fields

B. SCANNER / UNIVERSE / STEP OUTPUT DATA
This should appear instantly if already computed:
- bucket rank
- finalist selection state
- role
- score
- route/status
- scanner phase/step classification
- publication state
- updated timestamp
- freshness metadata
- known pending reasons
- file-written state

C. HISTORY-DERIVED DATA
This should fill when history is available:
- ATR
- baseline move
- volatility freshness
- bar-derived values
- history-dependent metrics

D. LIVE / TICK / SESSION DATA
This may legitimately wait:
- spread now
- spread median
- spread max
- SprATR
- tick age
- quote state
- session live-state
- live tradability fields

E. DOWNSTREAM STEP DATA
This should fill when its stage completes:
- correlation
- closest correlation symbol
- signal stack (future)
- downstream inference fields
- later trade-setup fields

Rule:
The HUD and dossier must make category state visible:
- available now
- pending history
- pending live quote
- pending downstream stage

============================================================
6. HUD TRUTH MODEL
============================================================

The HUD must show truth in this form:

Known Now:
- data already available in memory or broker specs
- should be displayed immediately

Pending:
- data not yet available
- must be labeled with exact reason

Pending reasons must be explicit, for example:
- PENDING_HISTORY
- PENDING_QUOTE
- PENDING_CORRELATION
- PENDING_BATCH
- PENDING_PUBLICATION
- PENDING_SESSION
- PENDING_RETRY
- PENDING_SYMBOL_INIT

The HUD must not show “0.00” without context when the real state is “pending”.
Prefer:
- PendingHistory
- PendingQuote
- PendingCorrelation
over misleading numeric zeroes

============================================================
7. SYMBOL COMPLETION MODEL
============================================================

Each symbol must eventually have a completion contract.

Suggested symbol state ladder:
- CREATED
- STATIC_READY
- SCANNER_READY
- HISTORY_READY
- LIVE_READY
- CORRELATION_READY
- COMPLETE

Equivalent simplified model:
- STATIC
- SCANNER
- HISTORY
- LIVE
- CORRELATION
- COMPLETE

Every active symbol should ultimately expose:
- CompletionState
- CompletionPercent
- PendingReasons
- FileWritten
- FileLastUpdate
- LastKnownCategoryReady states

Example:
- CompletionState: PARTIAL
- CompletionPercent: 62
- PendingReasons: HISTORY,QUOTE,CORRELATION

Rule:
Completion must reflect real readiness, not cosmetic fullness.

============================================================
8. SUGGESTED COMPLETION PERCENT WEIGHTING
============================================================

Completion percentages must be category-based and honest.

Initial suggested weighting:
- static/spec/identity data = 25%
- scanner/universe/selection data = 20%
- history-derived data = 20%
- live/tradability data = 20%
- correlation/downstream data = 15%

This can be revised later, but must remain consistent.

Example:
If static + scanner complete, history partial, live pending, correlation pending,
then symbol completion may be around 45%–65% depending on exact weighting logic.

Rule:
The percentage must help the operator understand reality quickly.

============================================================
9. BUCKET COMPLETION MODEL
============================================================

Each bucket must have an aggregate contract.

Per bucket, track:
- BucketName
- TargetCount
- WrittenCount
- CompleteCount
- PartialCount
- PendingCount
- CompletionPercent
- CurrentBatchState
- LastWriteTime
- LastWriteSymbol
- PendingReasonSummary

Bucket completion should roll up symbol-level truth.

Rule:
The operator must be able to glance at the HUD and know:
- which buckets are done
- which buckets are partially done
- which buckets are waiting
- why they are waiting

============================================================
10. TOP-5-PER-BUCKET PUBLICATION RULE
============================================================

Trader folder publication must reflect:
- top 5 symbols per bucket
- same rotation truth as the scanner
- no separate downstream re-ranking in the HUD

Operational rule:
- Step 9 decides the winners
- publication follows those winners
- HUD displays the same truth
- HUD must not invent alternate active sets

Performance note:
Because dossier writing is heavier, publication and refresh may proceed in batches of 5.
This is acceptable if:
- the rotation truth remains aligned
- the batch state is visible
- pending batch status is visible on HUD

============================================================
11. BATCH MODEL
============================================================

Batching is allowed for heavy dossier work.

Suggested batch principles:
- publish/refresh in bounded groups
- keep batch size fixed and visible
- record current bucket being processed
- record current symbol being processed
- record last file written
- record last write timestamp
- expose progress count on HUD

Example operator view:
- Buckets Active: 7
- Target Symbols: 35
- Files Written: 23
- Complete: 11
- Partial: 12
- Pending: 12
- Current Batch: 5/7
- Last File: HK_EQUITY_INDUSTRIALS -> 1800.HK

Rule:
Batching is acceptable only if visibility remains clear.

17. BUTTON / INTERACTION SAFETY RULES
============================================================

Buttons are currently fragile and must be handled cautiously.

Rules:
1. Do not combine full navigation overhaul with data-contract overhaul
2. First stabilize displayed truth
3. Then simplify button count
4. Then make bucket clicks reliable
5. Then add symbol drilldown
6. Then add signal detail view

Button system goals:
- fewer buttons
- larger click targets
- stable click state
- explicit current selection state
- no hidden overlapping click zones
- no dependence on unstable redraw timing
- no heavy recompute on button click

20. IMPLEMENTATION PRINCIPLE: HUD MUST NOT SCAN FILESYSTEM HEAVILY
============================================================

The HUD must not become a filesystem crawler.

Preferred data sources:
- runtime state already in memory
- publication index arrays
- active symbol arrays
- bucket aggregate arrays
- write-status fields updated during publication
- already known dossier state flags

Avoid:
- re-reading every dossier file every timer cycle
- re-computing bucket aggregates from disk every repaint
- expensive broker/spec calls every frame

Rule:
Maintain lightweight runtime mirrors for HUD display.

============================================================
22. READABILITY STANDARDS
============================================================

The operator has visibility constraints.
Therefore:

- text must remain large enough
- contrast must remain strong
- rows must be compact but readable
- avoid over-dense clutter
- prioritize summary truth over decorative detail
- use percentages and counts rather than long verbose text where possible
- present “done / partial / pending” clearly
- present per-bucket status in rows
- do not print every symbol file on Dev HUD
- highlight current work and last write clearly

============================================================
23. DEFINITIONS OF DONE / PARTIAL / PENDING
============================================================

DONE
- all required categories for current scanner target are ready
- file written
- no material pending blocker for current stage

PARTIAL
- file written
- some categories ready
- one or more pending categories remain

PENDING
- publication target exists
- file not yet fully ready
- major category not ready
- or symbol awaiting batch turn / quote / history / downstream stage

These definitions must be stable across the Dev HUD and the dossier headers.

26. SAFE INTEGRATION RULES
============================================================

The HUD overhaul must not:
- change startup order
- change universe loading truth
- change Step 8 truth
- change Step 9 truth
- change Step 10 truth
- re-route publication
- re-rank active symbols
- change file naming
- introduce heavy preflight logic
- poll disk aggressively
- block timer heartbeats
- break atomic write assumptions

The HUD overhaul may:
- consume new runtime visibility flags
- consume new completion fields
- consume bucket aggregates
- consume write-status markers
- consume batch-state markers
- consume pending reason counts

============================================================
27. HUD PROJECT DEPENDENCY RULE
============================================================

Any HUD enhancement must first answer:
- where does this value come from?
- is it already computed?
- is it safe to expose?
- is it cheap to expose?
- does it preserve scanner runtime safety?

If the value is not already computed or cheaply exposed,
do not force it into the HUD yet.

============================================================
28. IMMEDIATE RECOMMENDED START POINT
============================================================

The first practical implementation target should be:

1. Define symbol completion states
2. Define bucket aggregate states
3. Expose runtime write-status markers
4. Build Dev HUD bucket completion view
5. Keep Trader HUD minimal
6. Defer advanced buttons

This gives maximum truth visibility for minimum risk.

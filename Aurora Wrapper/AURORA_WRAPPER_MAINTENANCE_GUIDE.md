# AURORA WRAPPER MAINTENANCE GUIDE

## Purpose

This guide governs refresh, replacement, consolidation, and audit work for `Aurora Wrapper/`.
The wrapper is compiled canon, so meaningful changes must stay synchronized with active Blueprint truth and with Aurora office continuity surfaces.

## Safe refresh workflow

1. read office canon, task board, decisions, work log, SHA ledger, and the latest run
2. read kernel/settings/file-map plus the owning pack(s) being changed
3. read the relevant active Blueprint source-truth roots for that pack
4. decide whether the change belongs in an existing owning pack rather than a new wrapper file
5. update the smallest correct set of wrapper/governance/control files
6. refresh run continuity and SHA coverage when material active files changed
7. delete transitional wrapper companions once lossless consolidation is confirmed

## Research-guided wrapper design rules

- preserve exact doctrine where compact carry is possible; avoid vague summaries
- prefer richer owning packs over parallel pack-plus-manual structures
- preserve explicit missingness, refusal, and blocked-output law
- carry enough family/pattern/card/packet detail for wrapper-only operation without cloning the entire Blueprint
- keep authority order explicit so wrapper strength does not become sovereignty drift

## Replace-vs-edit law

- edit existing owning packs when doctrine belongs to their long-run boundary
- create a new wrapper file only when grouping, ambiguity reduction, and package usability clearly justify it
- treat temporary operating manuals and densification files as merge scaffolds, not permanent architecture by default
- if two wrapper files say nearly the same thing, consolidate into the clearer owner and delete the duplicate

## What stays outside wrapper hot path

- office work log and SHA ledger during normal operation
- append-only runs except for refresh/audit work
- extraction ledgers, supplements, queues, source sets, and passes
- archive mirrors and superseded generations
- planning artifacts that do not earn permanent wrapper residency

## Replacement checklist for future Codex/GPT passes

- confirm the owning pack still matches active Blueprint law
- answer what Blueprint knowledge is still under-carried before adding text
- answer whether any wrapper file is too thin, too summary-like, or too duplicate-heavy
- update settings/file-map/package-governance if wrapper shape changed
- remove transitional companions after lossless merge
- refresh office continuity and SHA ledger when covered active files changed materially

## Audit checklist for future passes

- authority order still explicit
- build-phase and execution-side locks still explicit
- bridge missingness/refusal law still explicit
- family-first and pattern-second doctrine still specific
- opportunity/deployability/geometry/card separation still explicit
- machine-safe vs human-only boundary still explicit
- packet/review/example distinction still explicit
- wrapper file count still within target range and below ceiling
- no manual/companion duplicates lingering without justification

## Current package posture after consolidation

The wrapper now targets a stable 13-file package:
- 3 router/metadata files
- 5 core doctrine hot-path packs
- 1 conditional support vault
- 4 maintenance/governance files

The temporary operating manuals were consolidation scaffolds and have been merged back into the owning packs.

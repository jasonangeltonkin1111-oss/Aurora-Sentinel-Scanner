# AURORA WRAPPER CONTROL PACK

## Wrapper-facing identity

Aurora is a family-first, pattern-second market-interpretation and execution-qualification system.
It consumes measured upstream context from ASC, interprets structural state honestly, preserves missingness explicitly, and only produces opportunity posture, geometry, and strategy cards after staged validation.
Aurora is still in a building phase, focused on execution-side logic rather than full production deployment.

## Authority order

1. `Aurora Blueprint/office/` active office files
2. active canonical control files (`AURORA_CONTROL_INDEX_V5.md`, `AURORA_OPERATOR_PROTOCOL.md`, `AURORA_RECOVERY_ORDER_V5.md`, `AURORA_PROGRESS_TRACKER_V6.md`)
3. active bridge and execution-side protocols
4. family/pattern doctrine and cards
5. this compiled wrapper package as a wrapper-facing canon derived from the above

## What must remain source-truth only

These stay primarily in `Aurora Blueprint/`:
- office control and checkpointing
- run history and continuity notes
- extraction bookkeeping, supplements, queueing, and deepening-pass logistics
- strengthening residue and lineage scaffolds
- historical generations and archive mirrors

## What is compiled here for wrapper use

- control/routing law relevant to wrapper operation
- active reading order and file routing
- building-phase and execution-side scope locks
- wrapper object/workflow chain
- enum and status distinctions needed during wrapper reasoning
- family and pattern selection logic
- compiled current family and pattern doctrine
- bridge intake expectations
- canonical example anchors

## Current system posture

- `BUILDING_PHASE` remains active: Aurora can formalize objects, packets, deployability, family/pattern competition, geometry, and card logic, but it should not over-claim production completeness.
- `EXECUTION_SIDE_ONLY` remains active: Aurora is building the interpretation/output side, not replacing ASC runtime measurement.
- active-vs-historical classification must stay explicit.
- lossless cross-linking is preferred over destructive consolidation.

## Wrapper operational laws

### 1. No fabricated upstream truth
If ASC context is missing, stale, degraded, or unavailable, preserve that state and downgrade downstream posture honestly.

### 2. Preserve layer order
Context intake → group context → market/execution interpretation → family competition → pattern competition → opportunity inventory → geometry → strategy card → EA-safe boundary.

### 3. Family doctrine is not generic strategy advice
Aurora families are canonical interpretation families with habitat, anti-habitat, competitor, trigger, and failure logic.
Do not flatten them into trend/range buzzwords.

### 4. Pattern doctrine depends on family context
Patterns are local structural roles downstream of family selection, not free-floating chart labels.

### 5. No hidden score logic
Competition, ranking, and eligibility may use structured reasons, but not synthetic weighted scores presented as certainty.

### 6. Machine-safe vs human-only boundary must stay explicit
Wrapper output may carry both, but they must not be conflated.

## Hot path vs cold path

### Hot path
Control pack, execution pack, family vault, pattern vault, bridge pack.

### Cold path
Office logs, SHA ledger, run history, extraction ledgers, supplements, queueing artifacts, strengthening residue, archives.

## Repo-based design judgment

1. Source-truth-only: office control, ledgers, runs, extraction bookkeeping, lineage residue.
2. Wrapper-facing canon: active control law, execution chain, family/pattern canon, bridge contract, anchored examples.
3. Must remain separate even in wrapper form: control vs execution vs family vs pattern vs bridge vs examples.
4. Safe grouping: many protocol/schema files can be compiled into a single execution pack; family files/cards can be compiled into one family vault; pattern atlas/cards can be compiled into one pattern vault.
5. Off hot path entirely: raw work log, SHA ledger, extraction queue, supplements, archive mirrors.
6. Summarize vs fully compile: control/logistics can be summarized; execution/family/pattern/bridge logic should be compiled with real distinctions preserved.
7. Wrapper direct reads: kernel, control pack, execution pack, family vault, pattern vault, bridge pack.
8. Maintenance-only reads: maintenance guide plus source office, runs, ledgers, doctrine roots.
9. Target hot-path count: 5 core packs plus kernel/settings/file map.
10. Best balance: one tiny router kernel, one routing map, one settings file, five major packs, one maintenance guide.

# AURORA BUILD PHASE LOCK

## PURPOSE

This file locks the current project posture.

Aurora remains in **building phase**.
It is not yet in full deployment phase, and it is not yet in full automation phase.

This lock exists because Aurora cannot advance safely beyond a certain point until ASC foundation truth is sufficiently complete and stable.

---

# 1. ROOT LAW

For now:
- ASC is the foundation
- Aurora is downstream of ASC
- Aurora can continue building its doctrine, object model, prompts, workflows, and pilot lanes
- Aurora cannot responsibly move into full live execution dependence or bounded automation dependence if ASC is still missing required upstream truth surfaces

Aurora may build.
Aurora may refine.
Aurora may harden.
Aurora may become more machine-usable.

But Aurora should not pretend ASC is already finished if it is not.

---

# 2. WHAT “BUILDING PHASE” MEANS

Building phase means the project should prioritize:
- doctrine completion
- wrapper hardening
- object-chain clarity
- worked examples
- pilot family lanes
- prompt refinement
- machine-safe field separation
- upstream ASC requirement visibility

Building phase does **not** yet mean:
- trusting the full system as finished
- treating every Aurora object as automation-ready
- pushing beyond the limits of current ASC truth

---

# 3. WHY THIS LOCK EXISTS

Aurora depends on ASC for measured world truth.

If ASC does not yet provide enough stable upstream surfaces, then Aurora will eventually hit a ceiling in:
- deployability accuracy
- session/timebox accuracy
- execution-surface accuracy
- opportunity ranking accuracy
- strategy-card trustworthiness

That means the correct posture is:
- keep building Aurora
- keep clarifying what Aurora needs from ASC
- do not confuse downstream sophistication with upstream completeness

---

# 4. WHAT AURORA CAN KEEP DOING NOW

Aurora can continue with:
- remaining honest book completion
- doctrine strengthening
- wrapper workflow hardening
- worked examples
- pilot family-lane packets
- generated strategy-card refinement
- review/diagnosis scaffolds
- machine-safe versus human-only boundary sharpening

These are still valid building-phase tasks.

---

# 5. WHAT AURORA SHOULD NOT CLAIM YET

Aurora should not claim yet that:
- ASC is finished
- the full battlefield is stable enough for final execution dependence
- the full wrapper is ready for broad automation
- the EA-safe subset is ready for large-scale live trust

Those claims must wait until the ASC foundation is further complete.

---

# 6. ASC DEPENDENCY CHECK

Aurora’s next ceiling points are especially tied to ASC progress in:
- session / timebox truth
- spread / friction truth
- execution continuity truth
- freshness / degradation truth
- shortlist / battlefield-selection truth
- any other upstream surface later required by deployability and geometry logic

If these are incomplete, Aurora remains in building phase by necessity.

---

# 7. PRACTICAL PROJECT POSTURE

The correct current posture is:
- **build Aurora as far as possible without faking ASC completeness**
- **use Aurora to make ASC requirements more explicit**
- **let Aurora become cleaner, sharper, and more machine-ready**
- **delay final execution dependence until ASC catches up where needed**

This is not a weakness.
It is the correct sequencing law.

---

# 8. CURRENT JUDGMENT

Aurora remains in building phase.

That means:
- continue building
- continue hardening
- continue making the system more usable
- but do not act as if the downstream layer can outrun the unfinished foundation

Aurora can keep evolving strongly.
It just cannot responsibly go beyond what ASC foundation truth can support.

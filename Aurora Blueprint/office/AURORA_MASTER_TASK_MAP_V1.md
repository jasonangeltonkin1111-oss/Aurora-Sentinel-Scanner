## MANDATORY VERSION + SHA DISCIPLINE

From this task-map version onward, every meaningful Aurora update should follow this rule set.

### 1. Version discipline
Material Aurora updates must do one or more of the following:
- bump the version/generation marker of the file when the file is versioned by generation
- update the internal status/version field when the file uses internal version metadata
- create the next run file when the pass is continuity-meaningful
- update wrapper status or task-map version when the compiled or control posture materially changes

Do not leave a materially changed active file looking unchanged in version posture.

### 2. SHA discipline
If any active file already covered by `Aurora Blueprint/office/SHA_LEDGER.md` changes materially, the SHA ledger must be refreshed in the same pass.

If a new file becomes active control truth or active canonical doctrine, decide explicitly whether it now belongs in SHA coverage.
Do not let the ledger drift silently.

### 3. Minimum continuity package for meaningful passes
At minimum, a meaningful Aurora pass should update:
- the actual target file(s)
- the relevant run record
- the SHA ledger when covered files changed materially
- the office/task/progress/control surface if the active picture changed materially

### 4. No fake freshness
Do not claim a control surface, task board, or ledger is current if repo state changed materially and those surfaces were not updated.

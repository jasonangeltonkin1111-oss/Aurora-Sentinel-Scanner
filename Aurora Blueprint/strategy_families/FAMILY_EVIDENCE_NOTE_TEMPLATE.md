# FAMILY EVIDENCE NOTE TEMPLATE

## PURPOSE

This file defines the standard structure for future Aurora family evidence notes.

Evidence notes are the layer below:
- family-definition files
- family cards
- strategy testing / evidence doctrine

They exist to record how a specific family becomes stronger, weaker, sharper, or less credible as:
- more books are processed
- more testing is done
- more data surfaces become available

This file is not a backtest report template.
It is the structured note format for accumulating family-specific evidence without losing continuity.

---

# 1. EVIDENCE NOTE DESIGN LAW

Every family evidence note must:
- stay attached to one family
- preserve both strengthening and weakening evidence
- keep source lineage explicit
- separate structural evidence from uplift evidence
- remain enrichable as more books and data arrive

Evidence notes must not become marketing summaries for a family.
They must preserve honest negative evidence too.

---

# 2. REQUIRED SECTIONS

Each family evidence note should contain these sections in order:

1. NOTE ID
2. FAMILY NAME
3. CURRENT STATUS SNAPSHOT
4. STRUCTURAL DISTINCTION UNDER REVIEW
5. NATIVE HABITAT EVIDENCE
6. ANTI-HABITAT EVIDENCE
7. PATTERN-CHILD EVIDENCE
8. DATA-SURFACE UPLIFT EVIDENCE
9. NEGATIVE OR WEAKENING EVIDENCE
10. OPEN QUESTIONS
11. FUTURE TESTING NEEDS
12. SOURCE LINEAGE
13. CURRENT JUDGMENT

---

# 3. WHAT EACH SECTION IS FOR

## NOTE ID
Stable note identifier such as:
- `FE-C01-001`
- `FE-C03-002`

## CURRENT STATUS SNAPSHOT
Quick status of the family at the time of the note.

Examples:
- BUILD_READY but still early
- BUILD_READY with weak anti-habitat clarity
- CONDITIONAL pending stronger session evidence

## STRUCTURAL DISTINCTION UNDER REVIEW
What exact distinction is being tested or sharpened?

Examples:
- Trend Continuation vs Breakout / Compression Release
- Failed Break / Trap Reversal vs unstable chop
- Balance Rotation vs early breakout acceptance

## NATIVE HABITAT EVIDENCE
What supports the family in the places it claims to belong?

## ANTI-HABITAT EVIDENCE
What supports suppressing the family where it does not belong?

## PATTERN-CHILD EVIDENCE
What pattern children sharpen the family meaningfully?

## DATA-SURFACE UPLIFT EVIDENCE
What extra data surfaces genuinely help?

Examples:
- ASC ranking
- live spread / liquidity conditions
- macro suppression / weighting
- session or timebox context

## NEGATIVE OR WEAKENING EVIDENCE
What weakens the family or shows that some distinction is softer than expected?

This section is mandatory.

## OPEN QUESTIONS
What remains unresolved?

## FUTURE TESTING NEEDS
What should be tested next if the family is to become more mature?

## SOURCE LINEAGE
Which files, books, passes, or notes support this evidence note?

---

# 4. EVOLUTION LAW

As more books are processed, family evidence notes should usually be updated by:
- adding stronger habitat evidence
- sharpening anti-habitat boundaries
- improving family-vs-family discrimination
- recording whether extra data surfaces actually help
- recording negative evidence honestly

Do not create a new family note every time unless the new evidence is clearly a separate question.

---

# 5. RELATIONSHIP TO BOOK EXTRACTION

New book extraction should usually feed family evidence notes only when it adds one or more of:
- stronger structural separation
- stronger anti-habitat clarity
- stronger pattern meaning
- stronger data-surface uplift insight
- stronger warnings against naive transfer or overfitting

This helps ensure the project keeps extracting real leverage instead of only adding more descriptive layers.

---

# 6. CURRENT JUDGMENT

This template exists so Aurora can accumulate family-specific evidence over time without losing continuity, without hiding negative evidence, and without turning the strategy layer into a purely descriptive archive.

# PRESERVED NON-CHART-NATIVE REGISTRY

## PURPOSE

This file preserves academically real strategy families that Aurora should not lose, but that do not belong in the first clean chart-native core family pack.

These are preserved because they may become important later once Aurora uses broader data surfaces such as:
- ASC internal feature outputs
- live execution / microstructure data
- cross-asset and spread relationships
- macro / regime information
- options and derivatives surfaces

This file exists so “not core now” does not become “forgotten forever.”

---

# 1. PRESERVED FAMILY LIST

## PRS-1. Pairs / Statistical Arbitrage Family

### Canonical idea
Exploit relative mispricing, spread behavior, or co-movement distortions across linked instruments.

### Why preserved
Too important to lose, but not a clean direct child of current single-chart Wave 1 state language.

### Why not core now
Requires spread construction, pair logic, or multi-series statistical framing.

### Likely data surfaces
- S2 — ASC Scanner / Internal Feature Surface
- S5 — Cross-Asset / Relative Surface

---

## PRS-2. Order-Book / Microstructure Tactics Family

### Canonical idea
Execution or directional tactics driven by live order-book, queue, or hidden-liquidity behavior.

### Why preserved
Real tactics, but not chart-only native.

## PRS-3. Options Overlay / Volatility Harvest Families

### Canonical idea
Systematic options-based or volatility-premium strategies such as buy-write / put-write / VRP-like overlays.

### Why preserved
Academically real and materially important, but outside the current Wave 1 core.

### Why not core now
Require options surface, contract logic, and stronger volatility/risk framework.

### Likely data surfaces
- S4 — Macro / Regime Surface
- options/derivatives extensions beyond current family layer

## PRS-4. HFT / Latency / Ultra-Short-Horizon Tactics

### Canonical idea
Strategies that depend on queue position, latency, microsecond reactions, or ultra-short-lived order-flow asymmetry.

### Why preserved
Real, but architecturally separate from current Aurora doctrine.

### Why not core now
Not compatible with the present chart-first family surface.

### Likely data surfaces


# AURORA OPPORTUNITY ABUNDANCE AND ANTI-STARVATION LAW

## PURPOSE

This file formalizes Aurora's abundance law.

Aurora already had an anti-starvation posture inside the opportunity inventory layer.
This file makes the broader doctrine explicit so future family growth, stage logic, packet design, and wrapper compilation do not quietly drift back toward narrowness.

---

# 1. ROOT LAW

Aurora should prefer truthful opportunity abundance over artificial scarcity.

That means:
- abundance is desirable when it comes from broader valid structure coverage
- starvation is undesirable because it hides real opportunity classes Aurora should be able to see
- quality must be preserved through structure, competition, filtering, and downgrade logic rather than through narrow doctrine
- multiple opportunities may coexist inside the same broader context
- observe-only opportunity is still part of the opportunity ecology
- no-trade remains a valid and sometimes necessary result

Aurora must never chase abundance by loosening standards blindly.
Aurora must earn abundance by expanding valid families, valid pattern variants, valid stage variants, and valid bucket-aware routing.

---

# 2. WHAT ABUNDANCE IS NOT

Aurora abundance is not:
- signal spam
- generic "setup everywhere" logic
- hidden threshold loosening
- family-blind pattern hunting
- symbol-bound opportunism
- forced early entry bias
- rewriting observe-only as tradeable just to avoid silence

If abundance is produced by lowering truth standards, it is not Aurora abundance.
It is drift.

---

# 3. ANTI-STARVATION OPERATING PRINCIPLES

Aurora should reduce starvation by doing the following:

1. preserve more than one live family candidate when competition is still honest
2. preserve more than one live pattern candidate when family posture still allows it
3. preserve observe-only and watch-only cases instead of discarding them too early
4. preserve structurally valid but timing-sensitive cases for later review or revisit
5. legitimize mid-progress, continuation, re-entry, and later-confirmation opportunity classes when the family still supports them
6. expand family and bucket coverage over time rather than pretending one small family set should explain the whole symbol universe
7. keep no-trade and structure-invalid outcomes available when the truth is genuinely weak or exhausted

---

# 4. ABUNDANCE FILTERING LAW

Aurora should grow opportunity count through layered truth:
- context truth
- family competition truth
- pattern competition truth
- opportunity ranking truth
- deployability truth
- geometry truth

This means the anti-noise filter lives in the architecture.
Aurora should not make the system "cleaner" by refusing to see valid opportunity classes upstream.
It should make the system cleaner by preserving structure-specific downgrade, invalidation, and no-card pathways downstream.

---

# 5. MULTI-OPPORTUNITY ECOLOGY LAW

One symbol or group-context may honestly contain:
- a primary live opportunity
- one or more secondary degraded opportunities
- one or more observe-only continuation or reversal paths
- a watch-only late-stage continuation case
- a salvage or recovery case that is structurally real but execution-sensitive

Aurora should preserve these distinctions instead of forcing every case into one trade/no-trade slot.

---

# 6. FAMILY-FIRST ABUNDANCE LAW

Abundance must remain family-first.

Aurora must not search for more opportunity by bypassing family competition and jumping straight to pattern labels.
The correct growth path is:
- more valid families
- more valid family branches
- better family stage language
- better family bucket targeting
- better downstream pattern differentiation inside those families

---

# 7. BUCKET-TARGETED ABUNDANCE LAW

Aurora should broaden opportunity coverage across a large symbol universe by becoming more bucket-aware, not more generic.

Correct direction:
- more family applicability detail by bucket and regime
- more environment-specific routing
- more family branches that are honest about where they belong and where they do not

Incorrect direction:
- pretending one family behaves the same in every bucket
- forcing all families to act as universal templates
- using abundance language to excuse weak routing

---

# 8. STAGE-AWARE ABUNDANCE LAW

Aurora must not equate opportunity only with the earliest possible entry point.

Some truthful opportunity classes are:
- early structure emergence
- developing structure with incomplete confirmation
- confirmed structure after additional evidence arrives
- mature continuation that still has usable path left
- late join with tighter evidence and smaller remaining asymmetry
- re-entry after a partial interruption inside still-valid family logic
- salvage / recovery when prior geometry failed but family structure did not fully die

A stage-aware system can be more abundant without becoming looser, because later stages often carry more evidence even when they carry less runway.

---

# 9. OBSERVE-ONLY AND NO-TRADE LAW

Observe-only is not failure.
It is part of the opportunity ecology when:
- structure is interesting but deployability is weak
- family logic is real but timing is poor
- the case needs more evidence
- the path may improve later

No-trade remains correct when:
- structure is invalid
- stage is exhausted beyond honest repair
- competition never resolved enough to justify preservation
- the project boundary makes execution dishonest

Aurora must preserve the difference.

---

# 10. CURRENT JUDGMENT

Aurora should now be developed as an abundance-seeking but truth-bounded opportunity ecology.

The system should become less starvation-prone by expanding real structural coverage, stage coverage, and bucket-targeted family breadth — not by becoming permissive.

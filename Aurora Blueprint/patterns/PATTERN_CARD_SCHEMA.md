# PATTERN CARD SCHEMA

## PURPOSE

This file defines the standard structure for Aurora pattern cards.

Pattern cards sit below:
- doctrine
- strategy families
- strategy family cards

They exist so patterns become:
- organized
- family-attached
- machine-translatable later
- enrichable as more books are processed

Pattern cards must never become a random textbook-shape dump.
They must stay attached to family meaning.

---

# 1. PATTERN DESIGN LAW

Every pattern card must be:
- attached to at least one parent family
- explicit about habitat and anti-habitat
- explicit about what it is commonly confused with
- honest about what is still provisional
- easy to deepen as more books are processed

Pattern cards must not pretend they are final implementation recipes.

---

# 2. REQUIRED CARD SECTIONS

Each pattern card should contain these sections in order:

1. PATTERN ID
2. PATTERN NAME
3. STATUS
4. PARENT FAMILY / FAMILIES
5. CANONICAL THESIS
6. PRIMARY MARKET HABITAT
7. PRIMARY STATE / SURFACE DEPENDENCIES
8. REQUIRED DATA SURFACES
9. HIGH-VALUE SUPPORTING DATA SURFACES
10. STRUCTURAL MEANING
11. STRUCTURAL INVALIDATION / FAILURE LOGIC
12. MAIN MISREADS / COMPETING INTERPRETATIONS
13. PATTERN REJECTION CONDITIONS
14. FUTURE TESTING / RESEARCH NEEDS
15. FUTURE ENRICHMENT PATH
16. LINEAGE
17. CURRENT JUDGMENT

---

# 3. STATUS VALUES

Allowed status values:
- `APPROVED_GROUP`
- `BUILD_READY_CHILD`
- `CONDITIONAL_CHILD`
- `PRESERVED_ONLY`
- `DEFERRED`

## Meaning
`APPROVED_GROUP`
- an approved atlas child group, but not yet decomposed into many narrower children

`BUILD_READY_CHILD`
- a clean, usable first-wave pattern child

`CONDITIONAL_CHILD`
- real and useful, but requires more doctrine or more data surfaces

`PRESERVED_ONLY`
- worth preserving, but not a current build target

`DEFERRED`
- known but intentionally not active yet

---

# 4. EVOLUTION LAW

As more books are processed, pattern cards should usually evolve by:
- sharpening meaning
- sharpening invalidation / failure logic
- clarifying misreads
- adding data-surface dependencies
- splitting broad groups into narrower children when justified

Pattern cards should rarely be rewritten from scratch unless the parent family logic changes materially.

---

# 5. CURRENT JUDGMENT

This schema exists so the pattern layer becomes a durable, organized child layer of the family system instead of turning into a loose pattern catalog.

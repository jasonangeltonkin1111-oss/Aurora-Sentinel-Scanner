# 2. PATTERN-LAYER STRENGTHENING ADDITIONS

## 2.1 Classical formations are valid only as family children

### Strengthening statement
Classical chart formations remain useful, but only when attached to a parent family and structural habitat.

### Why this is justified
The classical chart-reading sources contain many useful formations, but those formations become noisy if treated as standalone decision objects.

### Canonical implication
Aurora should continue preserving formations only through:
- family attachment
- habitat attachment
- anti-confusion boundaries

### What this changes
This validates the existing Pattern Atlas root law: patterns come after state, surface, and family.

---

## 2.2 Support / resistance meaning should enrich pattern cards directly

### Strengthening statement
Pattern cards should treat support/resistance as structural decision-zone meaning, not as decorative background context.

### Why this is justified
The uploaded price-action foundation uses support/resistance to explain why a formation matters, fails, or reverses.

### Canonical implication
Pattern interpretation should explicitly ask:
- is the structure forming at or through a meaningful decision zone?
- does that zone support continuation, rejection, reclaim, or rotational return?

### What this changes
This especially strengthens:
- Accepted Break And Hold
- Failed Break / Reclaim
- Range Rejection / Return To Value
- Compression Release

---

## 2.3 Continuation and reversal patterns need stronger failure logic

### Strengthening statement
A useful pattern is not defined only by how it appears when it works, but also by how it fails and what it becomes when it fails.

### Why this is justified
Both classical and quant-oriented sources become more valuable when patterns are treated as hypotheses with failure paths, not as guaranteed signals.

### Canonical implication
Pattern cards should continue preserving:
- invalidation logic
- competing interpretations
- rejection conditions

### What this changes
This strengthens the pattern-card architecture already created in the repo.

---

## 2.4 Visual pattern recognition and measurable pattern hypotheses should remain separate but linked

### Strengthening statement
A pattern can first be recognized visually, then translated later into measurable rule candidates.
These are related steps, not the same step.

### Why this is justified
The Harriman side is strongest as chart-reading and hypothesis generation.
The quant/Python side is strongest as rule formalization and testing.

### Canonical implication
Pattern cards should stay chart-first in meaning, while remaining open to later measurable translation.

### What this changes
This validates the current design where pattern cards are doctrinally readable now and machine-enrichable later.

---

## 2.5 Pattern groups should stay broad until the source base justifies narrower children

### Strengthening statement
Broad pattern groups should not be split into many narrow named children unless extraction or testing clearly shows that the narrower split adds real discrimination.

### Why this is justified
Bulk pattern catalogs and candle taxonomies can easily produce excessive fragmentation without enough real leverage.

### Canonical implication
Aurora should keep using:
- approved pattern groups first
- narrower child cards later only when justified

### What this changes
This protects the atlas from exploding into hundreds of weakly differentiated pattern files.

---

## 2.6 Quant/Python logic should later test pattern usefulness, not replace pattern meaning

### Strengthening statement
The quant lane should later test whether a pattern group adds discrimination or predictive value, but it should not erase the structural meaning that made the pattern worth preserving.

### Why this is justified
Quant books are best used to formalize and compare ideas, not to deny the value of prior chart-structure interpretation.

### Canonical implication
Pattern work should evolve in two linked stages:
1. preserve structural meaning
2. test measurable usefulness later

### What this changes
This gives the pattern layer a disciplined path toward Python/machine enrichment without turning the atlas into a purely statistical feature dump.
